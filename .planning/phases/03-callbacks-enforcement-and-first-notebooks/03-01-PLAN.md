    **Modify src/customer_service/tools/process_refund.py — two-step vetoable pattern:**

    Split `handle_process_refund` into three functions:
    1. `compute_refund_proposal(input_dict: dict, services: ServiceContainer) -> dict` — looks up customer, checks policy, returns a PROPOSAL dict with all fields (status, customer_id, order_id, amount, refund_id or rejection reason, policy_approved) but does NOT call `services.financial_system.process_refund()` yet. If customer not found, returns error dict.
    2. `commit_refund(proposal: dict, services: ServiceContainer) -> dict` — calls `services.financial_system.process_refund()` with the proposal data and returns the financial system result.
    3. `handle_process_refund(input_dict: dict, services: ServiceContainer) -> str` — calls compute_refund_proposal then commit_refund, returns JSON string. This preserves backward compatibility for non-callback code paths.

    The proposal dict must contain: `{"status": "proposed", "customer_id": ..., "order_id": ..., "amount": ..., "policy_approved": bool, "requires_review": bool, "customer_tier": str}`. This gives callbacks full context for their decision.

    **Modify src/customer_service/tools/handlers.py — extend dispatch():**

    New signature: `dispatch(tool_name: str, input_dict: dict, services: ServiceContainer, context: dict | None = None, callbacks: list | None = None) -> str`

    Logic changes in dispatch():
    1. Default context to `{}` if None, callbacks to `[]` if None.
    2. For process_refund SPECIFICALLY (the vetoable tool):
       a. Import and call `compute_refund_proposal(input_dict, services)` to get proposal dict
       b. Run each callback in callbacks list: `cb(tool_name, input_dict, proposal, context, services)`
       c. If any callback returns action="escalate": return callback.replacement (the escalation JSON)
       d. If any callback returns action="replace_result": use callback.replacement
       e. Otherwise: import and call `commit_refund(proposal, services)` and return json.dumps(result)
    3. For log_interaction (redaction before write):
       a. Run callbacks with input_dict and an empty result_dict BEFORE calling the handler
       b. If any callback returns action="replace_result": parse callback.replacement, update input_dict["details"] with redacted version, then call handler with modified input_dict
       c. Otherwise: call handler normally
    4. For all other tools (lookup_customer, check_policy, escalate_to_human):
       a. Call the handler first (pure reads or safe append-only)
       b. Parse the JSON result string into a dict
       c. Run callbacks: `cb(tool_name, input_dict, result_dict, context, services)`
       d. If any callback returns action="replace_result": return callback.replacement
       e. Otherwise: return original result string
    5. Existing error handling (unknown tool, KeyError/TypeError/ValueError) stays the same.

    Import additions at top of handlers.py:
    ```python
    from customer_service.tools.process_refund import compute_refund_proposal, commit_refund
    ```
